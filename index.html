01

Tipos de Procesadores

Microprocesador

Solo el CPU

CPU puro sin memoria ni periféricos integrados. Necesita placa base, RAM, chipset y chips de I/O para funcionar.

  • Ventajas Alta velocidad, multitarea, potencia de cómputo
  • Desventajas Alto costo, alto consumo energético, tamaño mayor
  • Ejemplos Intel Core i3/i5/i7/i9, AMD Ryzen, Intel Xeon
  • Uso PCs, laptops, servidores, workstations
Microcontrolador (MCU)

Sistema completo en un chip

Integra CPU + memoria Flash/SRAM/EEPROM + GPIO + ADC/DAC + UART/SPI/I2C. Autosuficiente, bajo consumo.

  • Ventajas Económico, compacto, arranque inmediato, sin SO
  • Desventajas Potencia y memoria muy limitadas
  • Ejemplos Arduino (ATmega), ESP32, PIC, STM32
  • Uso IoT, electrodomésticos, industria, wearables
SoC

System on Chip

Integración máxima: CPU + GPU + RAM + radio + I/O en un solo die de silicio. La línea con el MCU se difumina.

  • Smartphones Qualcomm Snapdragon, MediaTek Dimensity, Apple A-Series
  • Laptops Apple M-Series (SoC + memoria unificada)
  • Ventaja Latencia mínima entre componentes, menor consumo
//

Regla de oro Microcontrolador = autosuficiente para sistemas embebidos. Microprocesador = potencia máxima, necesita soporte externo. SoC = término amplio que engloba integración máxima, puede incluir MCU y microprocesador como subtipos.

Arquitecturas de núcleo más importantes

x86 / x86-64

CISC — dominante en escritorio y servidores

  • Origen Intel 8086 (1978), extendido a 64 bits por AMD (x86-64)
  • ISA CISC: instrucciones de longitud variable, muy complejas
  • Internamente Convierte instrucciones complejas en micro-ops tipo RISC para ejecutar con pipeline eficiente
  • Fortaleza Retrocompatibilidad total — software de décadas sigue corriendo
  • Consumo Mayor consumo energético; diseñado para alto rendimiento con disipación activa
  • Ejemplos Intel Core i3/i5/i7/i9, AMD Ryzen, Intel Xeon, AMD EPYC
  • Uso PCs, laptops, servidores, workstations, data centers

ARM

RISC — dominante en móvil, embebidos e IoT

  • Origen Acorn Computers (1985); hoy licenciado por ARM Holdings
  • ISA RISC: instrucciones simples de longitud fija, altamente optimizables
  • Modelo de negocio ARM no fabrica chips; vende licencias de diseño de núcleos (Cortex-M, Cortex-A, Cortex-R)
  • Fortaleza Eficiencia energética superior: máximo rendimiento por watt
  • Cortex-M Microcontroladores (STM32, nRF, RP2040) — sin SO, bare-metal
  • Cortex-A Procesadores de aplicación (smartphones, SBC, Macbooks)
  • Ejemplos Apple M-series, Qualcomm Snapdragon, Raspberry Pi, ESP32 (Xtensa, familia ARM)
  • Uso Smartphones, tablets, laptops, IoT, microcontroladores, servidores cloud
02

ISA — CISC vs RISC

Definiciones clave

ISA

Instruction Set Architecture

Interfaz entre el software y el hardware. Define qué instrucciones puede ejecutar el procesador, cómo se codifican y cómo interactúan con la memoria.

Microarquitectura

Implementación física de la ISA

Cómo se implementa internamente la ISA con transistores, cachés y lógica. Dos procesadores pueden compartir la misma ISA (x86) pero tener microarquitecturas distintas.

CISC

"Hace mucho en una sola instrucción"

  • Instrucciones Complejas, longitud variable, muchas operaciones por instrucción
  • Memoria Opera directamente sobre ella (no solo registros)
  • Pipeline Decodifica internamente en micro-operaciones (micro-ops)
  • CPI Variable — puede ser mayor a 1 por instrucción compleja
  • Registros Pocos registros de propósito general
  • Compilador Más simple — la CPU hace el trabajo
  • Ejemplo ISA x86, x86-64 (Intel, AMD)
  • Uso PC, laptops, servidores, software legacy

RISC

"Instrucciones simples, un ciclo de reloj"

  • Instrucciones Pocas, simples, longitud fija
  • Memoria Solo LOAD/STORE — todo lo demás opera en registros
  • Pipeline Diseñado nativamente para pipelining eficiente
  • CPI ~1 ciclo por instrucción
  • Registros Muchos registros de propósito general (ej: 32)
  • Compilador Más complejo — optimiza para usar instrucciones simples
  • Ejemplo ISA ARM, RISC-V, MIPS
  • Uso Smartphones, Macbook, IoT, microcontroladores

Punto de examen frecuente El CISC moderno también usa pipelining, descomponiendo instrucciones complejas en micro-ops internamente. En la práctica, Intel y AMD toman lo mejor de ambos mundos: ISA CISC por compatibilidad, pero ejecución interna tipo RISC.

03

Arquitecturas de Memoria

Von Neumann

Un solo bus para datos e instrucciones

  • Problema Cuello de botella de Von Neumann: no puede buscar instrucción y dato al mismo tiempo
  • Ciclo Fetch → Decode → Execute (secuencial)
  • Ventaja Simplicidad y flexibilidad: misma RAM para código y datos
  • Uso Microprocesadores, PCs, Raspberry Pi, Laptops

Harvard

Buses separados para datos e instrucciones

  • Ventaja Acceso simultáneo: mayor velocidad, sin cuello de botella
  • Memorias Dos memorias físicamente separadas: programa y datos
  • Desventaja Mayor complejidad de hardware, inflexibilidad
  • Uso Microcontroladores: Arduino (ATmega), PIC, DSPs
Harvard Modificada

El estándar en procesadores modernos

Núcleo que usa arquitectura Harvard internamente (caché L1 separada para instrucciones y datos), pero una sola memoria RAM externa. Combina velocidad de Harvard con la flexibilidad de Von Neumann.

  • L1-I: caché de instrucciones, separada (Harvard puro)
  • L1-D: caché de datos, separada (Harvard puro)
  • L2 unificada: instrucciones + datos compartidos (aquí se "modifica")
  • L3 compartida entre núcleos → RAM DDR externa

Memoria en microcontroladores

Flash

No volátil — el programa

Guarda el código compilado que se ejecuta. Persiste al apagar. Escritura lenta (solo al programar). En Arduino UNO: 32 KB.

SRAM

Volátil — variables en ejecución

Variables globales, locales, pila (stack). Se borra al apagar. Acceso muy rápido. En Arduino UNO: solo 2 KB. El recurso más escaso.

EEPROM

No volátil — configuraciones

Guarda datos que deben persistir entre reinicios (calibraciones, ajustes de usuario). Direccionable byte a byte. Escritura lenta, ~100K ciclos de escritura. Arduino UNO: 1 KB.

04

Pipeline — Ejecución por Etapas

//

¿Qué es el Pipeline? Técnica que divide la ejecución de una instrucción en etapas independientes que se solapan en el tiempo. En lugar de esperar a que una instrucción termine por completo antes de empezar la siguiente, múltiples instrucciones avanzan simultáneamente en etapas distintas. Resultado: mayor throughput sin necesidad de aumentar la frecuencia de reloj.

Sin Pipeline

Una instrucción a la vez, de inicio a fin

  • Comportamiento La CPU termina completamente cada instrucción antes de comenzar la siguiente
  • Ciclos Si cada instrucción tarda 4 ciclos, 3 instrucciones = 12 ciclos
  • Analogía Una sola persona lavando, enjuagando y secando cada plato antes de tomar el siguiente
  • Resultado Los recursos del hardware están ociosos la mayor parte del tiempo

Con Pipeline

Varias instrucciones en progreso al mismo tiempo

  • Comportamiento Mientras la instrucción 1 está en la etapa EX, la instrucción 2 ya está en ID y la instrucción 3 en IF
  • Ciclos Las mismas 3 instrucciones se completan en ~6 ciclos con solapamiento
  • Analogía Una línea de ensamblaje: lavado, enjuague y secado ocurren en paralelo con platos diferentes
  • Resultado Throughput cercano a 1 instrucción por ciclo de reloj
//

Ventajas del Pipeline Mayor rendimiento (throughput) sin cambiar la frecuencia de reloj. Mejor aprovechamiento de los recursos hardware. Escala bien combinado con otras técnicas como superescalaridad y ejecución fuera de orden. Es la base del rendimiento moderno tanto en RISC (ARM, RISC-V) como en CISC (x86 descompone instrucciones complejas en micro-ops internamente para poder hacer pipeline).

05

Paralelismo & Tipos de Arquitecturas Flynn

//

Power Wall (2005) Hasta ~2005 se aumentaba el rendimiento subiendo la frecuencia de reloj (GHz). El problema: la potencia dinámica P = α·C·V²·f escala linealmente con f, pero el calor generado no se podía disipar de forma efectiva. La solución fue cambiar de estrategia: menos GHz, más núcleos.

Tipos de paralelismo

ILP

Instruction-Level Parallelism

Múltiples instrucciones ejecutándose dentro de un solo hilo. Implementado con pipeline, ejecución fuera de orden (OOO), ejecución especulativa y superescalaridad.

TLP

Thread-Level Parallelism

Múltiples hilos corriendo en diferentes núcleos físicos al mismo tiempo. Base de los procesadores multinúcleo (Intel i7, Apple M-series).

DLP

Data-Level Parallelism

La misma operación aplicada a múltiples datos simultáneamente. Base de las GPUs y vectorización de código (SIMD). Fundamental en IA y gráficos.

Taxonomía de Flynn

SISD
Single Instruction, Single Data

Computación clásica pre-2005. Un hilo, un dato. PCs antiguas con un solo núcleo sin SIMD.

SIMD
Single Instruction, Multiple Data

Una instrucción opera sobre múltiples datos en paralelo. Base de GPUs e IA. Ejemplo: sumar 8 floats de 32 bits en un solo ciclo con instrucciones AVX en x86.

MISD
Multiple Instruction, Single Data

Múltiples instrucciones sobre un mismo dato. Teórico, rarísimo en práctica. Usado en sistemas de tolerancia a fallos (aeroespacial).

MIMD
Multiple Instruction, Multiple Data

El estándar moderno: múltiples núcleos, cada uno con su propio flujo de instrucciones y datos independientes. Intel, AMD, Apple M-series, servidores.

Jerarquía de caché compartida (arquitectura multinúcleo)

//

Coherencia de caché Con múltiples núcleos compartiendo L3 y RAM, puede haber inconsistencias si dos núcleos modifican el mismo dato. Se resuelve con protocolos de coherencia de caché que monitorean el estado de los datos entre núcleos a través del bus compartido.

Arquitecturas para IA / IoT / Edge

TPU

Tensor Processing Unit

Diseñado por Google para acelerar operaciones de matrices (multiplicaciones de tensores). Usa un array sistólico de unidades de multiplicación-acumulación (MAC).

NPU

Neural Processing Unit

Acelerador de IA integrado en SoC de consumo (Qualcomm Hexagon, Apple Neural Engine). Permite inferencia local en móvil sin enviar datos a la nube.

TinyML

IA en microcontroladores

Redes neuronales preentrenadas y cuantizadas corriendo en MCU. Reconocimiento de voz, gestos o imágenes completamente offline. Ej: TensorFlow Lite Micro.

Edge Computing

Cómputo cerca de la fuente

Procesar los datos en o cerca del sensor/dispositivo en lugar de enviar todo a la nube. Reduce latencia, mejora privacidad y ahorra ancho de banda. Ej: cámaras inteligentes, Alexa, vehículos autónomos.

06

Ley de Amdahl

Speedup = 1 / ( (1 − P) + P/N )
P porcentaje paralelizable del código (0 a 1)
N número de procesadores disponibles
(1−P) fracción serial — el techo absoluto del speedup

Ejemplo clásico del parcial: 95% paralelo, fracción serial = 5%, N → ∞

S = 1 / [(1 − 0.95) + 0.95/∞]
S = 1 / [0.05 + 0]
S = 1 / 0.05 = 20× — el máximo posible sin importar cuántos CPUs agregues.

Consecuencia práctica La Ley de Amdahl explica por qué simplemente agregar más núcleos no siempre escala. Apple Silicon usa núcleos P (Performance, optimizan la fracción serial) y núcleos E (Efficiency, manejan lo paralelo) — exactamente la respuesta correcta al problema que describe Amdahl.

Sistemas Embebidos

MCU — Microcontroller Unit

"Todo en un chip — sin sistema operativo"

  • Arquitectura Harvard (buses de programa y datos separados)
  • ISA RISC (AVR, ARM Cortex-M, PIC)
  • Software Bucle infinito directo, sin SO, interrupciones para eventos
  • Memoria Flash (código) + SRAM (datos) + EEPROM (persistencia)
  • Ejemplos Arduino UNO (ATmega328P), ESP32, STM32, PICAXE

SBC — Single Board Computer

"Placa madre miniatura — con Linux"

  • Arquitectura Von Neumann (bus unificado)
  • ISA RISC (ARM Cortex-A)
  • Software Linux completo, Python, múltiples procesos/hilos
  • Memoria RAM DDR + almacenamiento en tarjeta SD
  • Ejemplos Raspberry Pi 4 (4× Cortex-A72), Orange Pi, Banana Pi

08

Buses y Comunicación

Bus de Datos

Transporta la información

Bidireccional. Ancho: 8, 16, 32 o 64 bits. Define cuántos bits mueve la CPU en un ciclo. Un bus de 64 bits puede mover 8 bytes por ciclo.

  • Ancho A mayor ancho, más datos por ciclo — rendimiento directamente relacionado
  • x86-64 Bus de datos de 64 bits — mueve 8 bytes por ciclo de bus
  • ARM MCU Típicamente 32 bits; algunos Cortex-M0 con bus de 16 bits internamente
  • Ejemplo Si lees un int de 32 bits desde memoria, el bus de datos lleva esos 4 bytes en un solo ciclo (en un bus de 32+ bits)
Bus de Dirección

Indica origen o destino

Unidireccional (CPU → memoria o periférico). Indica la dirección de memoria a la que va o de la que viene el dato. N bits → 2ᴺ posiciones direccionables.

  • Fórmula Espacio de memoria = 2ᴺ bytes, donde N = ancho del bus de dirección
  • x86-64 48 bits efectivos → 256 TB de espacio de direcciones virtual
  • ARM Cortex-M 32 bits → 4 GB de espacio de direcciones (suficiente para MCU)
  • Ejemplo Un bus de dirección de 16 bits puede acceder a 2¹⁶ = 65,536 posiciones (64 KB)
Bus de Control

Sincroniza todo

Señales de lectura/escritura (R/W), reloj, interrupciones, bus request/grant. Coordina cuándo y cómo se transfiere la información entre los demás buses.

  • R/W Señal que indica si la operación es lectura (CPU ← memoria) o escritura (CPU → memoria)
  • CLK Señal de reloj que sincroniza las transferencias
  • IRQ / NMI Líneas de interrupción — señalan al CPU que un periférico requiere atención
  • Bus Request/Grant Permite a otros maestros del bus (DMA) tomar control del bus temporalmente
  • RESET Reinicia el estado de todos los componentes del sistema
//

Protocolos de comunicación en MCU UART (serial asíncrono, 2 hilos, simple), SPI (síncrono, 4 hilos, rápido, full-duplex), I2C (síncrono, 2 hilos, multi-dispositivo, más lento). Los tres son los más comunes en sistemas embebidos y examen.

Conceptos Clave

Rendimiento y métricas

MIPS

Millones de instrucciones por segundo

Métrica de rendimiento del procesador. Calculadora de MIPS abajo.

MIPS = f (MHz) / CPI
CPI

Ciclos por instrucción

Promedio de ciclos de reloj necesarios para ejecutar una instrucción. RISC ideal = 1 CPI. En la práctica, hazards elevan el CPI efectivo.

Mecanismos especiales

Interrupciones

Eventos prioritarios

Pausan el programa principal para atender un evento urgente. Más eficientes que Polling (que consulta constantemente y drena ciclos/batería).

  • CPU pausa hilo actual
  • Guarda contexto (PC, registros)
  • Salta a la ISR (rutina de servicio)
  • Al terminar, restaura contexto y continúa
Watchdog Timer

Guardián del sistema

Timer que cuenta hacia cero. Si el programa no lo reinicia ("patear al perro") antes de que expire, el sistema se reinicia automáticamente. Fundamental en sistemas de misión crítica donde un cuelgue es inaceptable.

Polling

Consulta activa y periódica

El CPU consulta repetidamente el estado de un periférico o evento en un bucle, esperando que cambie. Simple de implementar y predecible en tiempo, ya que el CPU siempre sabe cuándo revisará el estado.

  • Comportamiento El programa pregunta continuamente: "¿Ya llegó el dato?", "¿Está listo el periférico?"
  • Ventaja Simple, sin overhead de manejo de interrupciones, latencia determinista y baja en eventos muy frecuentes
  • Desventaja Desperdicia ciclos de CPU y batería cuando el evento es poco frecuente; el CPU no puede hacer otra cosa mientras espera
  • Uso típico Eventos de tiempo real muy frecuentes, protocolos que requieren respuesta inmediata, sistemas simples sin SO
  • Ejemplo Leer continuamente el pin de un botón dentro de un while(1) en un Arduino

Consumo energético

P_dinámica = α · C · V² · f
α factor de actividad — fracción de transistores que conmutan
C capacitancia total del chip
V² voltaje al cuadrado — el parámetro más impactante
f frecuencia (relación lineal)

Race to Sleep

Enciende al máximo, termina rápido, duerme

  • Idea Ejecuta tan rápido como sea posible para minimizar el tiempo activo total
  • Ventaja Menor energía si los periodos de sleep son largos y la tarea es corta
  • Uso Sensores IoT con batería que envían un dato y duermen

Slow & Steady

Reduce voltaje y frecuencia al mínimo suficiente

  • Idea Bajar V reduce P cuadráticamente — el mayor ahorro de energía
  • Ventaja Consumo constante y predecible para cargas de trabajo continuas
  • Uso Wearables, dispositivos de transmisión continua
